# 《交通事件文本结构化识别系统》技术路线

## 1. 技术路线概述

本项目采用“规则库 + 交通领域词典 + NLP 模型 + 大语言模型辅助校验”的融合技术路线，实现交通事件文本从非结构化表达到结构化数据的自动转换。

系统整体流程如下：

```text
多源交通文本
  ↓
文本清洗与预处理
  ↓
规则库与交通领域词典匹配
  ↓
事件分类与实体识别
  ↓
事件关系抽取
  ↓
结构化 JSON/Excel 输出
  ↓
大语言模型辅助校验
  ↓
数据库存储与可视化展示
```

## 2. 五层技术架构

### 2.1 数据来源层

数据来源层负责接入多类交通文本。

| 数据来源 | 示例 |
|---|---|
| 事故警情 | 今天上午8点30分，某路段发生两车追尾 |
| 道路巡查 | 巡查发现某路口信号灯故障 |
| 群众投诉 | 学校门口长期违停，影响通行 |
| 交通新闻 | 受暴雨影响，某快速路出现拥堵 |
| 舆情文本 | 网友反映某路段早高峰拥堵严重 |
| 模拟样本 | 项目团队根据真实场景构造的测试文本 |

### 2.2 文本处理层

文本处理层负责对原始文本进行标准化处理。

主要任务：

1. 去除无关符号和重复空格；
2. 将文本按标点进行分句；
3. 识别并标准化时间表达；
4. 提取候选地点、道路名和路口名；
5. 识别数字、车辆数量和伤亡人数；
6. 为后续规则匹配和模型识别提供统一输入。

### 2.3 智能识别层

智能识别层是系统核心，完成事件分类、实体识别和关系抽取。

| 子任务 | 方法 |
|---|---|
| 事件分类 | 关键词规则、交通领域词典、文本分类模型 |
| 时间识别 | 正则表达式、时间归一化规则 |
| 地点识别 | 道路后缀规则、行政区词典、地点实体识别模型 |
| 车辆识别 | 车辆词典、数量规则 |
| 伤亡识别 | 伤亡表达正则规则 |
| 道路影响识别 | 占道、封闭、缓行、排队等规则 |
| 拥堵程度识别 | 基于关键词的等级判定 |
| 关系抽取 | 事件模板、句式规则、后续模型扩展 |
| 大模型校验 | 对低置信度样本进行字段补全和一致性检查 |

### 2.4 结构化存储层

结构化存储层负责将识别结果转化为统一格式。

输出格式：

1. JSON：用于接口传输和系统展示；
2. Excel：用于人工查看、统计和申报材料展示；
3. 数据库：用于历史记录查询、人工校正和可视化分析。

核心表：

- traffic_event：交通事件主表；
- event_entity：事件实体表；
- event_relation：事件关系表；
- correction_record：人工校正记录表。

### 2.5 应用展示层

应用展示层面向用户提供系统功能。

主要页面：

1. 文本识别页面；
2. 结构化结果展示页面；
3. 人工校正页面；
4. 历史记录查询页面；
5. 统计分析页面；
6. 后续地图可视化页面。

## 3. 核心识别流程

### 3.1 事件分类

系统首先根据关键词和领域词典判断事件类型。

示例规则：

| 关键词 | 事件类型 |
|---|---|
| 追尾、碰撞、剐蹭、侧翻 | 交通事故 |
| 拥堵、缓行、排队、车流缓慢 | 道路拥堵 |
| 施工、养护、抢修、占道施工 | 道路施工 |
| 管制、封闭、限行、禁止通行 | 交通管制 |
| 信号灯故障、井盖缺失、护栏损坏 | 设施故障 |
| 暴雨、大雾、积水、结冰 | 恶劣天气影响 |
| 违停、逆行、占道、闯红灯 | 违法行为 |

后续可训练文本分类模型，将规则分类升级为“规则 + 模型融合分类”。

### 3.2 实体识别

实体识别任务用于抽取交通事件核心字段。

| 字段 | 初期方法 | 后续方法 |
|---|---|---|
| 时间 | 正则表达式 | 时间识别模型 |
| 地点 | 道路后缀规则、行政区词典 | NER 模型、地图地名库 |
| 车辆 | 车辆词典、数量规则 | NER 模型 |
| 伤亡 | 正则表达式 | 规则 + 模型 |
| 道路影响 | 关键词规则 | 事件要素抽取模型 |
| 拥堵程度 | 关键词分级 | 分类模型 |

### 3.3 关系抽取

关系抽取用于将实体连接成完整事件结构。

示例：

```text
追尾事故 - 发生时间 - 今天上午8点30分
追尾事故 - 发生地点 - 重庆市南岸区学府大道
追尾事故 - 涉及车辆 - 两车
追尾事故 - 造成后果 - 1人轻伤
追尾事故 - 影响道路 - 占用右侧车道
```

初期可采用模板规则实现，后续可扩展为关系抽取模型。

## 4. 大语言模型辅助校验方案

大语言模型不作为初期唯一识别方式，而作为辅助模块，用于提升复杂文本处理效果。

适用场景：

1. 规则识别置信度较低；
2. 文本较长且包含多个事件；
3. 群众投诉文本口语化严重；
4. 字段之间存在冲突；
5. JSON 输出缺少必填字段。

校验内容：

| 校验项 | 说明 |
|---|---|
| 字段补全 | 根据上下文补充遗漏字段 |
| 类型检查 | 判断事件类型是否合理 |
| 关系一致性 | 检查地点、伤亡、道路影响是否归属于同一事件 |
| JSON 格式修正 | 保证输出符合 JSON Schema |
| 低置信度提示 | 标记需要人工复核的字段 |

## 5. MVP 实现路线

MVP 阶段优先完成可运行原型，避免过度依赖复杂模型。

### 5.1 MVP 功能

1. 支持输入一段中文交通事件文本；
2. 使用规则匹配识别事件类型；
3. 使用正则表达式抽取时间、车辆数量、伤亡情况；
4. 使用道路后缀规则抽取地点；
5. 使用关键词规则判断道路影响和拥堵程度；
6. 输出标准 JSON；
7. 预留大模型接口。

### 5.2 MVP 代码结构

```text
traffic_event_structurer/
  __init__.py
  llm_client.py      # 大模型接口预留
  pipeline.py        # 识别流水线
  rule_engine.py     # 规则匹配核心
  schemas.py         # 字段定义与 JSON Schema
main.py              # 命令行入口
test_samples.py      # 测试样例
```

### 5.3 MVP 技术栈

| 模块 | 技术 |
|---|---|
| 开发语言 | Python |
| 规则抽取 | re 正则表达式 |
| 领域词典 | Python 列表/JSON 配置 |
| 数据输出 | JSON |
| 测试样例 | Python 脚本 |
| 后续接口 | FastAPI |
| 后续数据库 | SQLite/MySQL |
| 后续前端 | Vue/React |

## 6. 模型升级路线

### 6.1 阶段一：规则系统

目标是快速实现可演示原型。

重点：

- 构建交通事件关键词词典；
- 编写时间、地点、车辆、伤亡规则；
- 输出 JSON 结构化结果；
- 建立测试样例集。

### 6.2 阶段二：传统 NLP 模型

目标是提高事件分类能力。

可选方案：

- jieba 分词；
- TF-IDF 特征；
- SVM、朴素贝叶斯、逻辑回归分类器；
- 使用准确率、召回率、F1 值评估效果。

### 6.3 阶段三：深度学习模型

目标是提高复杂文本中的实体抽取能力。

可选方案：

- BERT 文本分类；
- BERT-CRF 命名实体识别；
- RoBERTa、MacBERT 等中文预训练模型；
- 使用标注数据进行微调。

### 6.4 阶段四：大模型辅助校验

目标是提升复杂文本和低置信度结果处理能力。

可实现能力：

- 结构化结果校验；
- 字段补全；
- 多事件拆分；
- JSON 格式修正；
- 人工复核建议。

## 7. 实验验证路线

实验任务：

1. 事件类型识别；
2. 实体抽取；
3. 关系抽取；
4. JSON 格式正确率评估；
5. 人工处理时间节省率评估。

评价指标：

| 指标 | 含义 |
|---|---|
| 准确率 | 系统识别结果中正确结果所占比例 |
| 召回率 | 标准答案中被系统识别出的比例 |
| F1 值 | 准确率和召回率的综合指标 |
| JSON 格式正确率 | 输出符合 JSON Schema 的比例 |
| 时间节省率 | 系统辅助相比纯人工整理节省的时间比例 |

对比方法：

1. 人工规则方法；
2. 通用大模型直接抽取方法；
3. 本项目融合方法。

## 8. 申报书表述版本

本项目采用规则库、交通领域词典、NLP 模型和大语言模型辅助校验相结合的技术路线，构建面向交通事件文本的结构化识别系统。系统首先对事故警情、道路巡查、群众投诉和交通新闻等多源文本进行清洗、分句和标准化处理；然后利用交通领域词典和规则模板识别事件类型、时间、地点、车辆、伤亡、道路影响等核心要素；在数据积累后，进一步引入文本分类模型和命名实体识别模型提升复杂文本识别能力；对于低置信度结果和长文本场景，引入大语言模型进行辅助校验、字段补全和 JSON 格式检查。最终，系统将识别结果转换为统一的结构化 JSON 或 Excel 数据，并支持数据库存储、人工校正和可视化分析展示。该技术路线兼顾大学生团队的实现可行性和项目后续扩展空间，适合作为创新创业训练项目开展研发与成果展示。
